# Project Recovery Guide

Use this when a project was generated by an older harness version or by an agent that wrote shallow docs.

## Step 1: Scaffold Missing Files

```bash
blueprint init --directory . --yes --merge
```

This adds missing harness files without replacing existing docs.

## Step 2: Explain Failures

```bash
blueprint explain-fail --directory .
```

Use the printed repair checklist as the work queue.

## Step 3: Upgrade Product Truth

Rewrite these first:

1. `docs/product/product-passport.yaml`
2. `docs/product/prd.md`
3. `docs/product/data-api-contract.md`
4. `docs/product/integration-protocol.md`
5. `docs/specs/state-machines.yaml`
6. `docs/specs/rbac.yaml`
7. `docs/specs/error-codes.yaml`

## Step 4: Upgrade Planning Artifacts

Then update:

1. `docs/EDGE_CASE_MATRIX.md`
2. `docs/TRACEABILITY_MATRIX.md`
3. `docs/TEST_MATRIX.md`
4. `docs/stories/*.md`
5. `docs/readiness-review.md`

## Step 5: Run Gates

```bash
blueprint lint --directory . --ci
blueprint extension run before_readiness --directory .
blueprint readiness --directory . --ci
```

## Common Repairs

| Failure | Repair |
| --- | --- |
| Missing spec files | Run init merge, then fill specs with project-specific state/RBAC/error contracts. |
| `risk_level: high-risk` drift | Use either `high` or `high-risk`, but sync `.blueprint/status.json` risk to the same meaning. |
| Story missing production sections | Regenerate story with `blueprint new-story`, then migrate old content into the new sections. |
| Boolean test matrix | Replace yes/no with scenario IDs, commands, fixtures, expected evidence, owner, and status. |
| Simulated research | Run reference/domain research and attach source inventory, evidence cards, claim map, and assumptions. |

## Stop Rule

Do not start implementation just because documents exist. Start only when lint and readiness prove the documents are deep enough.
