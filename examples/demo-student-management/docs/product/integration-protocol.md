# Integration Protocol: Student Management

## Idempotency Keys
Imports and bulk creates use `source_file_id + row_number + student_code`. Interactive create requests use a request id generated by the UI.

## Retry Policy
Interactive writes do not auto-retry after validation failure. Import jobs retry transient database or storage failures up to three times with exponential backoff.

## Signature Validation
No external webhook is in MVP. If a payment provider is added for tuition, every callback must validate provider signature before state changes.

## Callback Handling
No payment callbacks in MVP. Future tuition callbacks must be idempotent and reconcile against current invoice state.

## Dead Letter Handling
Failed import rows move to an import error report with row number, error code, and staff-visible recovery note.

## Reconcile Runbook
Operations staff can rerun an import in dry-run mode, compare failed rows, and reprocess only corrected rows.

## Observability
Create, import, update, and delete actions write audit logs with actor, role, entity id, and result.

## Security
All write commands require an authenticated admin or staff role. Viewer role cannot create or mutate student records.

## Test Requirements
Integration tests must cover duplicate code, missing required fields, unauthorized role, and import partial failure.
